Project: ECG Graph Extraction and Analysis System 
1. Introduction 
1.1 Purpose 
To develop a software system that can process ECG graph images, interpret the 
waveform, and extract clinically relevant values (e.g., heart rate, R-R intervals, PQRST 
durations). 
1.2 Scope 
The system will: 
• Accept ECG graph images as input. 
• Preprocess and digitize the graph. 
• Detect and extract key ECG features. 
• Output extracted values in a structured format. 
1.3 Definitions, Acronyms, and Abbreviations 
• ECG: Electrocardiogram 
• SRS: System Requirement Specification 
2. Overall Description 
2.1 User Needs 
• Clinicians and researchers need to automate the extraction of ECG parameters 
from scanned or digital ECG graphs. 
• The system should minimize manual measurement errors and speed up data 
analysis. 
2.2 Assumptions and Dependencies 
• Input images are standard 12-lead ECGs with clear gridlines. 
• The system will run on Windows/Linux with Python or MATLAB support. 
3. System Features and Requirements 
3.1 Functional Requirements 
3.1.1 Input Handling 
• The system shall accept ECG graph images in common formats (JPEG, PNG, 
PDF). 
• The system shall validate image quality and provide feedback if the image is 
unsuitable. 

3.1.2 Preprocessing 
• The system shall enhance image quality (denoising, contrast adjustment). 
• The system shall detect and correct image skew or rotation. 
3.1.3 Graph Digitization 
• The system shall detect gridlines and calibrate time and amplitude axes. 
• The system shall trace the ECG waveform accurately. 
3.1.4 Feature Extraction 
• The system shall identify and mark P , Q, R, S, and T waves. 
• The system shall calculate R-R intervals, heart rate, and other relevant metrics. 
• The system shall export results in CSV or JSON format. 
3.1.5 User Interface 
The system shall provide a simple GUI or command-line interface for input and 
output.3.1.5 User Interface (Detailed)

High-Level System Architecture Overview 
1. Architectural Style 
The system follows a modular, pipeline-based 
architecture with clear separation between: 
• Input & Presentation Layer 
• Processing & Analysis Layer 
• Comparison & Visualization Layer 
• Persistence (Database & Storage) Layer 
This separation ensures: 
• Scalability and maintainability 
• Independent evolution of UI, analysis algorithms, 
and storage 
• Traceability and data integrity (critical for medical 
data)

or\How ECG Delta Visualization Works 
Delta visualization answers the question: 
“How has this ECG changed compared to 
another ECG?” 
The system does this by aligning, normalizing, 
subtracting, and visually highlighting differences 
between two ECGs in a way that is clinically 
interpretable. 
1. Inputs to Delta Visualization 
The comparison always starts with two ECGs: 
• ECG-A (baseline / earlier ECG) 
• ECG-B (follow-up / later ECG) 
Each ECG may come from: 
• A newly uploaded image, or 
• A stored ECG image from the database (original 
image preserved). 
The system uses: 
• The original ECG image for display and 
verification 
• The digitized waveform (already extracted in your 
pipeline) for computation 

2. Pre-Processing & Normalization (Critical Step) 
Raw ECG images cannot be compared directly. Before 
computing differences, the system performs 
normalization so that both ECGs are on a common 
scale. 
2.1 Grid Calibration 
From your existing digitization step: 
• Time axis → milliseconds per pixel 
• Amplitude axis → millivolts per pixel 
This ensures: 
1 mm = 40 ms 
1 mm = 0.1 mV 
(or the detected equivalent) 
2.2 Lead Selection (if applicable) 
• If multiple leads exist, the system compares: 
o Same lead in both ECGs (e.g., Lead II vs 
Lead II) 
• Lead mismatch blocks comparison and triggers a 
warning. 
3. Temporal Alignment (Very Important) 
Two ECGs represent different heartbeats, so they must 
be aligned in time. 
The system supports automatic alignment using one of 
the following: 
3.1 R-Peak Alignment (Primary) 
• Detect R-peaks in both signals 
• Select a representative R-peak (e.g., first stable 
beat) 
• Shift ECG-B so its R-peak aligns with ECG-A 
This ensures: 
• QRS complexes sit on top of each other 

• PR, ST, and T-wave comparisons are meaningful 
3.2 Cross-Correlation (Fallback) 
• Used if R-peak detection is noisy 
• Maximizes waveform similarity to find best 
alignment shift 
4. Delta Computation (Numerical Core) 
After alignment, the system computes point-wise and 
feature-wise differences. 
4.1 Waveform Delta (Sample-Level) 
For each time sample t: 
Delta(t) = ECG_B(t) − ECG_A(t) 
This produces a delta waveform showing: 
• Positive deflection → increase in amplitude 
• Negative deflection → decrease 
4.2 Feature-Level Delta (Clinically Meaningful) 
The system also computes deltas for extracted ECG 
features: 
Feature Delta Example 
Heart Rate +12 bpm 
R-R Interval −80 ms 
PR Interval +20 ms 
QRS Duration No change 
QT / QTc +35 ms 
ST Elevation +0.15 mV 
Each delta is labeled as: 
• Increased 
• Decreased 

• Unchanged (using configurable clinical 
thresholds) 
5. Delta Visualization (What the User Sees) 
5.1 View 1: Overlay Comparison (Primary Clinical View) 
• ECG-A plotted in blue 
• ECG-B plotted in red 
• Shared grid for direct comparison 
• Feature markers (P , Q, R, S, T) can be toggled 
ON/OFF 
✅ Best for: 
• Shape changes 
• QRS widening 
• ST-segment shifts 
5.2 View 2: Delta Waveform (Difference View) 
• Single waveform showing ECG_B − ECG_A 
• Zero line at baseline 
Visual interpretation: 
• Flat line → no change 
• Spikes around QRS → conduction changes 
• Sustained deviation → ST/T changes 
✅ Best for: 
• Subtle changes not obvious in overlay 
• Quantifying magnitude of change 
5.3 View 3: Highlighted Change Regions 
The system highlights waveform sections where: 
• |Delta| exceeds a clinically relevant threshold 
Example: 
• ST segment ≥ ±0.1 mV deviation → highlighted in 
orange/red 
• QT prolongation → highlighted duration box 

✅ Best for: 
• Fast clinical interpretation 
• Decision support 
6. Summary Delta Panel (Numbers with Context) 
Alongside the plots, the system displays: 
• Metric 
• ECG-A value 
• ECG-B value 
• Delta 
• Status (↑ ↓ =) 
Example: 
Heart Rate:        68 → 82 bpm   (+14 ↑) 
PR Interval:       160 → 180 ms  (+20 ↑) 
QRS Duration:      94 → 96 ms    (+2 =) 
ST Deviation:      0.02 → 0.18mV (+0.16 ↑) 
7. Traceability & Database Integration 
When comparison is performed: 
• The system logs: 
o Record IDs of both ECGs 
o Alignment method used 
o Delta metrics 
• Comparison results can be exported (CSV/JSON) 
• Original images remain unchanged and 
retrievable 
8. Why This Design Is Clinically Sound 
This approach: 
• Preserves original ECGs (legal/medical integrity) 
• Compares signals only after proper calibration 
• Aligns heartbeats, not just timestamps 
• Separates visual inspection from quantitative 
delta 
• Scales from research use → clinical review

AR-2.2.3\ The architecture shall support future extensibility, including additional ECG

3.1.6 ECG Comparison & Delta Visualization 
3.1.6.1 Overview 
The system shall support comparison of two ECGs and provide delta visualization, 
enabling users to identify and assess changes between ECG recordings over time.\ The 

comparison shall include both visual and numerical representations of differences 
after proper normalization and alignment of ECG signals. 
3.1.6.2 Input Selection 
FR-3.1.6.2.1\ The system shall allow the user to select two ECGs for comparison from 
the following sources: 
• two newly uploaded ECG images, or 
• one uploaded ECG image and one stored ECG record, or 
• two stored ECG records from the database. 
FR-3.1.6.2.2\ Each selected ECG shall be associated with: 
• its original ECG image, and 
• its digitized waveform derived from the preprocessing and digitization pipeline. 
3.1.6.3 Pre-Comparison Normalization 
FR-3.1.6.3.1\ The system shall normalize both ECGs to a common scale prior to 
comparison using grid-based calibration derived during digitization: 
• time axis calibration (milliseconds per pixel), and 
• amplitude axis calibration (millivolts per pixel). 
FR-3.1.6.3.2\ Where multiple leads are present, the system shall compare only 
matching leads between the two ECGs.\ If a lead mismatch occurs, the system shall 
prevent comparison and notify the user. 
3.1.6.4 Temporal Alignment 
FR-3.1.6.4.1\ The system shall temporally align the two ECG waveforms before delta 
computation. 
FR-3.1.6.4.2\ The system shall support at least one of the following alignment methods: 
• R-peak alignment (primary method), or 
• waveform cross-correlation (fallback method). 
FR-3.1.6.4.3\ The selected alignment method shall ensure that corresponding cardiac 
cycles are aligned to allow meaningful comparison of ECG features. 
3.1.6.5 Delta Computation 
3.1.6.5.1 Waveform-Level Delta 
FR-3.1.6.5.1.1\ After alignment, the system shall compute a waveform-level delta 
signal defined as the point-wise difference between the two ECG waveforms: 

Delta(t) = ECG_B(t) − ECG_A(t) 
FR-3.1.6.5.1.2\ The delta signal shall retain the same time and amplitude units as the 
normalized ECGs. 
3.1.6.5.2 Feature-Level Delta 
FR-3.1.6.5.2.1\ The system shall compute deltas for extracted ECG features, including 
at minimum: 
• heart rate, 
• R-R interval statistics, 
• P wave duration (if available), 
• PR interval, 
• QRS duration, 
• QT / QTc interval (if available), 
• ST deviation metrics (if supported). 
FR-3.1.6.5.2.2\ For each feature, the system shall calculate: 
• value from ECG-A, 
• value from ECG-B, 
• numerical delta, 
• delta classification (increase, decrease, or no change). 
FR-3.1.6.5.2.3\ Delta classification thresholds shall be configurable or based on 
clinically relevant default thresholds. 
3.1.6.6 Delta Visualization 
FR-3.1.6.6.1 Overlay Visualization 
The system shall present an overlay view showing: 
• ECG-A and ECG-B plotted on the same calibrated grid, 
• distinct visual styles (e.g., colors) for each ECG, 
• optional visualization of detected ECG features (P , Q, R, S, T points). 
FR-3.1.6.6.2 Delta Waveform Visualization 
The system shall display a delta waveform view that: 
• plots the computed delta signal against time, 
• includes a zero baseline reference, 
• visually distinguishes positive and negative deviations. 
FR-3.1.6.6.3 Highlighted Change Regions 

The system shall highlight waveform regions where the absolute delta exceeds 
predefined thresholds, including but not limited to: 
• ST-segment deviations, 
• prolonged or shortened intervals. 
FR-3.1.6.6.4 Interaction Controls 
The visualization shall support: 
• zooming and panning, 
• toggling gridlines and markers, 
• switching between overlay and delta views. 
3.1.6.7 Delta Summary Panel 
FR-3.1.6.7.1\ The system shall display a summary panel presenting feature-level 
comparison results, including: 
• ECG-A value, 
• ECG-B value, 
• delta value, 
• direction indicator (increase, decrease, unchanged). 
3.1.6.8 Data Integrity & Traceability 
FR-3.1.6.8.1\ The system shall not modify the original ECG images during the 
comparison process. 
FR-3.1.6.8.2\ The system shall record comparison metadata, including: 
• identifiers of both ECG records, 
• alignment method used, 
• computed delta metrics. 
FR-3.1.6.8.3\ The system shall allow export of delta analysis results in CSV or JSON 
format. 
3.1.6.9 Acceptance Criteria (Suggested) 
• User can select two ECGs for comparison. 
• The system produces: 
o an overlay visualization, 
o a delta waveform visualization, 
o highlighted change regions, 
o a numerical comparison summary. 
• Results can be exported without altering original data.

• Upload Image Section: A button to browse and upload ECG graph files. 
• Graph Display Area: Shows the original ECG image with detected PQRST points 
marked. 
• Results Panel: Displays extracted values like heart rate, R-R interval, and 
durations. 
• Export Button: Allows saving results in CSV or JSON format.
